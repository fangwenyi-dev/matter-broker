# Matter-MQTT Broker 使用说明

> ESP32-S3 智能开窗器 LoRa 网关桥接器
> 将 LoRa 子设备桥接到 Matter 生态（Apple HomeKit / Google Home / Samsung SmartThings / 涂鸦）

---

## 一、项目概述

本项目运行在 ESP32-S3-WROOM-1-N16R8（16MB Quad Flash + 8MB Octal PSRAM）上，实现：

1. **内置 MQTT Broker**（mosquitto v2.0.20，端口 1883）—— 接收 LoRa 网关上报的设备状态
2. **协议桥接层**—— 将 $SH MQTT 协议与 Matter 属性双向转换
3. **Matter Bridge**—— 为每个 LoRa 子设备动态创建 WindowCovering 虚拟端点（Lift + Tilt 双滑块），接入 Apple HomeKit / Google Home / Samsung SmartThings / 涂鸦 App

### 架构数据流

```
[LoRa 子设备（开窗器）]
        ↓ LoRa 无线
[LoRa 网关]
        ↓ MQTT over WiFi ($SH 协议)
        ↓ broker 地址: matter-broker.local:1883
[ESP32-S3（内置 MQTT Broker + Matter Bridge）]
        ↓ 协议桥接层（$SH ↔ Matter 属性双向转换）
[Matter WindowCovering 虚拟端点（Lift + Tilt + 电池）]
        ↓ Matter over WiFi (CASE Session)
[Apple HomeKit / Google Home / Samsung SmartThings / 涂鸦]
```

### 设备角色

| 设备 | 角色 | 说明 |
|---|---|---|
| ESP32-S3 N16R8 | Matter Bridge + MQTT Broker | 运行 mosquitto broker（端口 1883）接收 LoRa 网关上报，同时作为 Matter Bridge 将子设备暴露给各生态 App |
| LoRa 网关 | 协议转换中继 | 连接 LoRa 子设备，通过 MQTT（$SH 协议）与 ESP32 通信；支持 mDNS 域名解析，可通过 `matter-broker.local` 连接 ESP32 |
| LoRa 子设备（开窗器） | 受控终端 | 通过 LoRa 无线连接网关，上报位置/电池/状态，接收控制命令（升降/停止/内倒）|

### 位置语义反转（重要）

| 协议 | 0 | 100 |
|---|---|---|
| LoRa | 关闭 | 打开 |
| Matter WindowCovering | 0% = 完全打开 | 100% = 完全关闭 |

转换公式：`matter_percent = 100 - lora_value`

## 三、按键操作

| 操作 | 功能 | 说明 |
|---|---|---|
| 2 击 | 启动 LoRa 配对模式 | 对所有在线网关发送配对命令 |
| 3 击 | 删除所有 LoRa 子设备 | 遍历所有网关下的子设备，逐个发送解绑命令（间隔 200ms） |
| 5 击 | 重置 Matter | **仅**重置 Matter 配网信息，不清除 WiFi |
| 长按 5 秒 | 重置 WiFi | **仅**清除 WiFi 凭证，不重置 Matter |

> 硬件约束：5 击与长按严格分离，5 击不会触发 WiFi 重置，长按不会重置 Matter。

## 四、配网流程

### 1. 首次使用

1. 烧录固件后，设备启动并打印：
   ```
   按键: 2击=LoRa配对, 3击=删除所有子设备, 5击=重置Matter, 长按5s=重置WiFi
   等待 LoRa 网关上线（自动绑定 + 发现设备）
   使用 Apple Home / Google Home 扫描配网二维码进行配对
   ```
2. 串口会打印 Matter 配网二维码：
   ```
   I (xxxx) app_matter_bridge: ===== Matter 配网信息（请用 Apple Home / Google Home 扫码）=====
   SetupQRCode: [MT:Y.K9042C00KA0648G00]
   Copy/paste the below URL in a browser to see the QR Code:
   https://project-chip.github.io/connectedhomeip/qrcode.html?data=MT%3AY.K9042C00KA0648G00
   Manual pairing code: [34970112332]
   ```
   - **SetupQRCode**：复制 URL 到浏览器查看二维码图片
   - **Manual pairing code**：11 位数字手动配对码（二维码无法扫描时用）
3. 在手机 App（Apple HomeKit / Google Home / Samsung SmartThings / 涂鸦）中扫描 Matter 配网二维码
4. App 通过 BLE 与设备配网，配置 WiFi 凭证
5. WiFi 连接成功后，设备自动：
   - 禁用 WiFi 省电模式（WIFI_PS_NONE，防止 App 离线）
   - 启动内置 MQTT Broker
   - 启动本地 MQTT 客户端（纯发布命令）
   - 等待 LoRa 网关连接

> ⚠️ **重要警告**：本固件已禁用 softAP，正常情况下设备**不会**开启任何 WiFi 热点。Matter 配网的唯一正确方式是扫描串口打印的二维码，通过 BLE 传 WiFi 凭证。

### 2. LoRa 网关接入

LoRa 网关需配置连接到本设备，端口 1883，认证 admin/admin。连接地址支持两种方式：

#### 方式 A：mDNS 域名（推荐）

本设备启用 mDNS 服务，WiFi 连接成功后注册 hostname `matter-broker`，LoRa 网关可通过 `matter-broker.local` 域名解析本设备 IP。

**工作机制（单 mDNS 实例，espressif/mdns）**：

本项目使用 espressif/mdns（IDF 组件）作为唯一 mDNS 实例，负责全部 mDNS 功能：
- **Matter DNS-SD 服务发布**（SRV/TXT/AAAA 记录）：让 Apple Home / Google Home / 涂鸦 App 能发现设备
- **`matter-broker.local` A 查询响应**：让 LoRa 网关能解析到 ESP32 的 IP

main.cpp 在 `app_matter_bridge_start()` 之前调用 `mdns_init()` + `mdns_hostname_set("matter-broker")`，确保 espressif/mdns 的 hostname 不被覆盖。

> **重要**：`CONFIG_USE_MINIMAL_MDNS=n` 是单实例方案的关键配置。不要改回 `CONFIG_USE_MINIMAL_MDNS=y`，否则 minimal mdns 会与 espressif/mdns 抢占 UDP 5353 端口，导致 Matter 服务发布失败（报错 `Failed to initialize advertiser: 3000008`）。

```
LoRa 网关配置：
  Broker Address: matter-broker.local   (mDNS 域名，IP 变更不影响)
  Broker Port: 1883
  Username: admin
  Password: admin
  订阅主题: gateway/{gw_sn}/req
  发布主题: gateway/rpt_rsp
```

**前提**：LoRa 网关必须支持 mDNS 域名解析（Linux/OpenWrt 系统原生支持，嵌入式 RTOS 需确认 lwIP 配置）。

**验证**：PC 测试域名是否生效
```bash
# Windows（需启用 Bonjour 服务）
ping matter-broker.local

# macOS/Linux 原生支持
ping matter-broker.local
```

串口日志会打印：
```
I (xxxx) main: mDNS hostname 已设置: matter-broker.local
```

#### 方式 B：IP 地址

若 LoRa 网关不支持 mDNS 解析，使用 IP 地址：

```
LoRa 网关配置：
  Broker IP: <ESP32的局域网IP>  (通过 DHCP 获得，从串口日志查看)
  Broker Port: 1883
  Username: admin
  Password: admin
  订阅主题: gateway/{gw_sn}/req
  发布主题: gateway/rpt_rsp
```

> **建议**：在路由器后台给 ESP32 的 MAC 地址绑定固定 IP，避免 DHCP 重分配导致 IP 变更。

网关上报 001（绑定）后，本设备自动注册网关（不再创建网关 Matter 端点，避免 HomeKit 中出现虚拟设备）。
网关上报 002（设备列表）后，本设备自动为每个子设备创建 WindowCovering 端点。

### 3. 动态添加设备

**支持**：Matter 协议支持动态端点，已配网的控制器会订阅 Aggregator 的 `PartsList`，新端点创建后自动通知控制器。

**各 App 行为**：
- Apple HomeKit / Google Home / Samsung SmartThings：新设备几秒内自动出现
- 涂鸦 App：可能需要下拉刷新、退出重进、或彻底关闭 App 后重新打开

> **Samsung SmartThings 兼容性**：SmartThings 对 Matter Bridge 的动态端点支持完善，新设备添加后自动出现在「所在房间」列表中。WindowCovering 的 Lift/Tilt 双滑块在 SmartThings 中渲染为「窗户」设备类型，支持位置拖动和停止按钮。

**端点创建后自动初始化**：
- Reachable=true（强制初始化 data version + 触发属性报告，确保 App 显示在线）
- ConfigStatus=0x19 + Type=0x08（显式覆盖 NVS 旧值，确保滑块正确渲染）
- Lift/Tilt 位置初始值 = 0（完全打开/无倾斜，LoRa 上报后更新真实值）
- Lift/Tilt data version 初始化（确保控制器首次读取能获得值）
- HardwareVersion=1 + SoftwareVersion=1（HomeKit 设备详情页显示）
- VendorName + VendorID + ProductName + ProductID（BDI 簇认证属性，继承根节点认证状态）
- BatVoltage/BatPercentRemaining/BatChargeLevel 初始为 null（等待 LoRa 首次上报）
- 跳过 SDK 默认 delegate 触发的首次 TargetPosition 变更（per-feature 独立，2s 超时失效）

### 4. 配网失败排查

| 串口日志 | 含义 | 解决方法 |
|---|---|---|
| `WiFi STA 断开, reason=201` | 找不到 SSID | 检查配网时输入的 WiFi 名称 |
| `WiFi STA 断开, reason=15` | WiFi 密码错误 | 重新配网，输入正确密码 |
| `WiFi STA 断开, reason=2` | 认证超时 | 检查路由器信号，靠近路由器重试 |
| `WiFi 重连失败已超 20 次` | 累计重连 20 次失败 | 自动清除 WiFi 凭据重启进入配网模式 |
| 一直看不到 `SetupQRCode` | 二维码未打印 | 确认固件是修复后版本 |

### 5. 重新配网（更换 WiFi）

1. **长按 BOOT 按钮 5 秒以上** → 清除 WiFi 凭证，设备重启
2. 重新使用 Apple Home / Google Home 配网

> 清除 WiFi 不影响 Matter 配网状态，Apple Home 中的设备不会消失，但需要重新选择 WiFi。

## 五、MQTT 协议（$SH）

### 连接参数

| 参数 | 值 |
|---|---|
| Broker 端口 | 1883（CONFIG_BROKER_PORT） |
| 用户名 | admin（CONFIG_MQTT_BROKER_USERNAME） |
| 密码 | admin（CONFIG_MQTT_BROKER_PASSWORD） |
| QoS | 1 |

> ⚠️ **安全警告**：默认 admin/admin 为弱密码，broker 监听 0.0.0.0 暴露在局域网。生产部署务必通过 `idf.py menuconfig` 修改为强密码。

### 主题结构

| 主题 | 方向 | 说明 |
|---|---|---|
| `gateway/{gw_sn}/req` | 本设备 → 网关 | 下发控制/配对命令 |
| `gateway/rpt_rsp` | 网关 → 本设备 | 网关上报状态/设备列表 |

### 消息类型（ctype）

| ctype | 名称 | 方向 | 说明 |
|---|---|---|---|
| 001 | 网关绑定 | 网关→本设备 | 自动回复 errcode=0，记录网关 SN，请求设备列表 |
| 002 | 设备列表 | 网关→本设备 | 解析 devices 数组，为每个设备创建 Matter 端点 |
| 003 | 设备配对/解绑 | 双向 | 添加/删除子设备，通过 bind 字段区分（1=配对，0=解绑） |
| 004 | 设备控制 | 本设备→网关 | Matter 属性变更转发为控制命令 |
| 005 | 状态上报 | 网关→本设备 | 双格式：直接字段 / attrs 数组 |

### 003 配对/解绑响应处理

LoRa 网关的 003 响应可能不包含 `bind` 字段，此时通过设备是否已存在于映射表来推断操作类型：
- 设备已在映射表中 → 推断为解绑响应（bind=0）
- 设备不在映射表中 → 推断为配对响应（bind=1）

此逻辑确保 3击删除所有子设备时，网关回复被正确识别为解绑响应而非配对响应。

### 005 状态上报字段

直接字段格式：
- `position` / `r_travel`：位置（0-100，>100 表示未校准/离线标记，会被忽略）
- `battery` / `voltage`：电池电压原始值（÷10 = 伏特，如 105 → 10.5V）
- `state`：设备状态（0=关闭，1=打开）

> battery 与 voltage 同时存在时，voltage 优先；值不一致时打印告警日志。
> battery/voltage 兼容字符串和数字类型。原始值超出 80-140 范围会被丢弃（防溢出）。

attrs 数组格式：attribute 名 `voltage` 同样表示电池原始值，`r_travel`/`state` 同样有范围校验。

完整 $SH 协议参考 HA 集成项目：
`e:\AI\huijian-gateway\ha-window-controller-gateway\custom_components\window_controller_gateway\`

## 六、Matter 集成

### 端点结构

```
Node (endpoint 0)
└── Aggregator (endpoint 1)
    ├── 子设备1（WindowCovering Lift+Tilt + PowerSource + BDI）
    ├── 子设备2
    └── ...（最多 12 个，CONFIG_MAX_BRIDGED_DEVICES）
```

> 每个开窗器子设备创建 1 个 Matter 端点。电池电压和状态作为 cluster 内嵌在端点中（PowerSource = 电池，WindowCovering = 状态），不额外占用端点配额。

### Cluster 说明

| Cluster | 端点 | 功能 |
|---|---|---|
| BridgedDeviceBasicInformation | 子设备 | Reachable（在线/离线）、VendorName、VendorID、ProductName、ProductID、SerialNumber、HardwareVersion、SoftwareVersion |
| WindowCovering | 子设备 | Lift 升降位置控制 + Tilt 内倒控制 + StopMotion（停止=LoRa 101） |
| PowerSource | 子设备 | BatVoltage（毫伏）+ BatPercentRemaining + BatChargeLevel（满/低/危） |

> **网关虚拟设备已移除**：网关不再创建 Matter 端点，HomeKit 中只显示真实的 LoRa 子设备。
> 配对模式通过 2击按键触发，删除所有子设备通过 3击按键触发。

### MFi 认证（BDI 簇属性）

HomeKit 对动态添加的桥接设备会检查 BDI 簇的 VendorID/ProductID 进行认证校验。端点创建时自动添加以下属性，使桥接设备继承根节点的认证状态：

| 属性 | 值 | 说明 |
|---|---|---|
| VendorName | 成都慧尖科技有限公司 | HomeKit "生产企业" |
| VendorID | CONFIG_DEVICE_VENDOR_ID | MFi 认证检查（与根节点一致） |
| ProductName | 设备显示名 | HomeKit 首页图标下方文字 |
| ProductID | CONFIG_DEVICE_PRODUCT_ID | MFi 认证检查（与根节点一致） |
| SerialNumber | LoRa 设备 SN | HomeKit "序列号" |
| HardwareVersion | 1 | HomeKit "固件版本"（硬件） |
| SoftwareVersion | 1 | HomeKit "固件版本"（软件） |

> 缺失 VendorID/ProductID 时，HomeKit 显示"此配件尚未经过认证可与homekit配合使用"。

### WindowCovering 配置

| 属性 | 值 | 说明 |
|---|---|---|
| Type | 0x08 (TiltBlindLiftAndTilt) | 升降+倾斜型，HomeKit 渲染 Lift + Tilt 双滑块 |
| EndProductType | 0x00 (RollerShade) | 卷帘型，HomeKit 渲染位置滑块 + 开/关/停止按钮 |
| feature_flags | 0x17 | Lift + PositionAwareLift + Tilt + PositionAwareTilt |
| config_status | 0x19 | Operational + LiftPositionAware + TiltPositionAware |
| Lift 初始位置 | 0 (0%=完全打开) | 初始为打开状态，LoRa 上报后更新真实值 |
| Tilt 初始位置 | 0 (0%=无倾斜) | 初始为未内倒状态 |

> **Type 选择说明**：原 0x00 (RollerShade) 仅支持升降，HomeKit 不渲染 Tilt 滑块。改为 0x08 (TiltBlindLiftAndTilt) 后 HomeKit 正确渲染 Lift + Tilt 双滑块。EndProductType 规范枚举无"开窗器"值，0x00 (RollerShade) 的 UI 交互（位置滑块 + 按钮）与开窗器完全一致。

### Reachable 属性（关键）

控制器通过 `BridgedDeviceBasicInformation` 簇的 `Reachable` 属性判断设备在线/离线。

**关键机制**：端点创建后调用 `app_matter_bridge_update_reachable()` 执行两步操作：
1. `attribute::update()` 更新存储值（值未变化时被 SDK 跳过）
2. `MatterReportingAttributeChangeCallback()` 强制标记属性为脏，触发报告下发

此举绕过 SDK 的值变化检查，确保控制器首次读取时能获得 data version，避免设备显示离线。

**去重优化**：在设备表中记录 `last_reachable` 状态，仅当值真正变化或首次调用时才触发报告。避免网关频繁上下线时产生报告风暴（N 个设备 × M 次报告）。

### CASE Session 建立后的端点同步

端点在 CASE Session 建立之前创建时，`endpoint::enable()` 的 PartsList 通知会丢失。控制器首次建立 CASE Session 后：

1. 通知 aggregator 的 PartsList 属性变更（触发控制器重新发现端点）
2. **不批量通知 Reachable**：HomeKit 初始属性读取和订阅初始报告已覆盖 Reachable 值，无需额外推送

**CASE Session 去重**：30 秒内重复的 CASE Session 建立事件跳过 PartsList 通知。HomeKit 从后台恢复或网络抖动会频繁重建 CASE Session，30 秒窗口覆盖正常重连间隔，减少流量尖峰，缩短"正在更新"时间。

### 电池电压转换

- LoRa 原始值（如 105）→ 毫伏：`voltage_mv = 原始值 × 100`（105 → 10500mV = 10.5V）
- 有效范围：原始值 80-140（对应 8V-14V，12V 锂电池正常工作范围 9.5V-12.6V）
- 百分比自动估算：原始值 80-140 线性映射为 0-100%
- BatChargeLevel 自动映射：0-20% → Critical（危急），21-70% → Warning（警告），71-100% → Ok（正常）
- HomeKit 根据 BatChargeLevel 显示电池图标（满/低/危），不更新会导致图标始终显示"满"

### Matter 事件

| 事件 | 触发 | 动作 |
|---|---|---|
| LIFT_CHANGED | App 拖动 Lift 滑块 | 发送 004 控制命令（位置反转后下发） |
| STOP_MOTION | App 点停止 | 发送 LoRa 101（暂停），500ms 内跳过 target 变更 |
| TILT_TOGGLE | App 拖动 Tilt 滑块 | 发送内倒命令（value=200），同步 Current=Target |
| COMMISSIONING_COMPLETE | Matter 配网完成 | 向所有已注册网关请求设备列表（002），加速端点创建 |

> **网关配对/删除功能**已从 Matter OnOff 改为按键触发：
> - 2击按键 → 启动所有在线网关的 LoRa 配对模式
> - 3击按键 → 删除所有 LoRa 子设备（逐个发送解绑命令，间隔 200ms）

## 七、网关离线检测

- 系统监控 task 每 30 秒检查一次
- LoRa 网关超过 900 秒（15 分钟）无任何消息上报 → 标记为离线，打印告警：
  ```
  LoRa 网关离线（900s 无消息）: <网关SN>
  ```
- 网关离线时同步更新该网关下所有子设备的 Matter Reachable=false
- 网关恢复上报时自动重新标记为在线，同步子设备 Reachable=true
- **不删除**离线网关的 Matter 端点，避免 App 中设备频繁消失/出现
- **离线超 1 小时**：清理网关表项（释放槽位供新网关使用），但**保留**设备→网关映射表，确保网关恢复时能更新子设备 Reachable
- 线程安全：`s_gateways` 和 `s_device_gateway_map` 跨 task 访问用 `taskENTER_CRITICAL` 保护
- **快照模式**：临界区内收集所有匹配设备 SN 到栈数组，退出临界区后逐个调用 Matter API，避免"临界区退出-重入"导致 for 循环跳过相邻匹配项

## 十三、常见问题（FAQ）

### Q1: BLE 配网时报 `BLE_INIT: Malloc failed`

**原因**：NimBLE 内存占用内部 RAM 过多。
**解决**：确认 sdkconfig.defaults 已配置：
- `CONFIG_BT_NIMBLE_MEM_ALLOC_MODE_EXTERNAL=y`
- `CONFIG_BT_NIMBLE_MAX_CONNECTIONS=1`
- `CONFIG_SPIRAM_MALLOC_ALWAYSINTERNAL=4096`
- MQTT 队列用 `xQueueCreateWithCaps(MALLOC_CAP_SPIRAM)`

### Q2: 002 设备列表解析失败

**原因**：消息超过缓冲区大小。
**解决**：已将 data 缓冲设为 4096 字节，支持 12 设备列表（约 900 字节）。

### Q3: 设备列表出现 002 响应死循环刷屏

**原因**：broker 对所有 PUBLISH 回调，本地客户端发布的 002 响应被自身再次处理。
**解决**：已在 `handle_mqtt_message` 入口按 `client_id == s_bridge_sn` 过滤自身消息。

### Q4: 电池电压显示异常（如 4.4V）

**原因**：网关故障发送异常值（如 700），×100 后溢出 uint16_t。
**解决**：已加范围校验（80-140），超出丢弃并打印告警。

### Q5: battery 字段解析失败

**原因**：网关可能以字符串形式发送 battery（如 `"105"`），原代码只接受数字。
**解决**：已用 `parse_number_field` 兼容字符串和数字类型，并检查 strtol 溢出。

### Q6: 设备在涂鸦 App 显示离线（在线设备: 0台）

**原因**：BridgedDeviceBasicInformation 簇的 Reachable 属性 data version 未初始化。
**解决**：已在端点创建后调用 `app_matter_bridge_update_reachable()` 强制初始化 data version + 触发属性报告。

### Q7: 添加设备时开窗器意外动作

**原因**：SDK 默认 delegate 在端点创建后设置默认 TargetPosition，触发属性回调，导致立即下发 LoRa 控制命令。
**解决**：已添加 `just_added_lift`/`just_added_tilt` 标志（per-feature 独立），2000ms 超时失效，跳过首次变更。

### Q8: 网关离线后 App 仍显示在线

**原因**：网关离线时未同步更新 Matter Reachable 属性。
**解决**：已在 `check_gateway_offline` 中同步更新 Reachable=false，网关恢复时同步 Reachable=true。

### Q9: 修改 WiFi 模式后无法配网

**原因**：sdkconfig 缓存旧配置。
**解决**：`idf.py fullclean` 后全量重编译。

### Q10: Nullable.h 编译错误

**原因**：esp_matter 版本为 main 分支。
**解决**：锁定 idf_component.yml 中 esp_matter 版本为 1.5.0，删除 managed_components 重新编译。

### Q11: 涂鸦 App 添加设备时显示"设备未认证"

**原因**：Matter 设备的 VID（厂商 ID）未在 CSA 官方认证列表（DCL）中注册。本项目默认使用测试 VID=0xFFF1。
**影响**：不影响配网流程、设备控制功能。仅可能影响涂鸦 App 的云端场景联动等高级功能。
**解决方案**（按成本递增）：
1. **方案 A（推荐，零成本）**：在涂鸦 App 添加流程中选择"添加未认证设备"或"其他 Matter 设备"，同意风险提示后即可正常使用。
2. **方案 B**：改用乐鑫已认证的 VID=0x1316 + 正式 DAC 证书。
3. **方案 C**：向 CSA 注册厂商会员 + 完成产品认证 + 烧录自有 DAC。

### Q12: HomeKit 动态添加设备显示"配件尚未经过认证"

**原因**：动态添加的桥接设备 BDI 簇缺少 VendorID 和 ProductID 属性，HomeKit 无法进行 MFi 认证校验。
**解决**：已在端点创建时手动添加 `create_vendor_id()` 和 `create_product_id()` 属性，使用与根节点相同的 VID/PID（来自 sdkconfig），使桥接设备继承根节点认证状态。

### Q13: LoRa 网关无法连接 broker（IP 变更后失效）

**原因**：ESP32 通过 DHCP 获取 IP，路由器重启或租约到期后 IP 可能变更。
**解决方案**（三选一）：
1. **mDNS 域名（推荐）**：本设备已启用 mDNS，LoRa 网关配置 broker 地址为 `matter-broker.local`。LoRa 网关自身也支持 mDNS 解析，无需手动输入 IP。
2. **路由器静态绑定**：在路由器后台给 ESP32 的 MAC 地址绑定固定 IP。
3. **ESP32 静态 IP**：代码中配置固定 IP（不推荐，换网段需重烧固件）。

### Q14: 网关离线/恢复时部分子设备状态未同步

**原因**：临界区"退出-重入"模式导致 for 循环 `j++` 跳过相邻匹配项。
**解决**：已改为快照模式——临界区内收集所有匹配设备 SN 到栈数组，退出临界区后逐个调用 Matter API。

### Q15: 编译报错 "Hash of the file does not match expected hash"

**原因**：`managed_components/` 下的组件被手动修改过，idf 检测到文件 hash 与记录的不匹配。
**解决**：将修改过的组件从 `managed_components/` 移到 `components/` 目录：

```powershell
# 以 espressif__mosquitto 为例
Move-Item -Path "managed_components\espressif__mosquitto" -Destination "components\espressif__mosquitto" -Force
```

> **注意**：`espressif__mosquitto` 必须移到 `components/`（含自定义认证回调代码）。`espressif__esp_matter` 已在 `components/` 目录（含 hostname 覆盖补丁）。

### Q16: 网关离线超 1 小时恢复后，子设备永久显示离线

**原因**：网关离线 1 小时后清理了设备→网关映射表，网关恢复时无法从映射表查找子设备更新 Reachable。
**解决**：已删除清理设备映射表的逻辑，仅清理网关表项（释放槽位）。映射表保留无害，网关恢复后 002 会刷新映射。

### Q17: r_travel=255 覆盖有效 position 导致位置不更新

**原因**：直接字段 r_travel 缺少范围检查，255（未校准标记）覆盖有效 position（如 80）。
**解决**：已添加 `<= 100` 范围检查，与 attrs 数组分支和 handle_ctype_002 保持一致。

### Q18: OTA 升级后设备自动回滚

**原因**：启用了 OTA 回滚保护但未在启动成功后标记新固件有效。
**解决**：已在 WiFi 连接成功后调用 `esp_ota_mark_app_valid_cancel_rollback()`。

### Q19: 3击删除子设备后端点残留

**原因**：网关 003 响应不包含 `bind` 字段，原代码缺省为 `bind=1`（配对），导致解绑响应被误判为配对响应。
**解决**：已增加推断逻辑——当响应缺少 `bind` 字段时，通过设备是否已存在于映射表来区分：已存在→解绑（bind=0），不存在→配对（bind=1）。

### Q20: HomeKit 打开 App 时长时间显示"正在更新"

**原因**：CASE Session 建立时批量通知所有端点的 Reachable 属性，产生大量属性报告与 HomeKit 初始读取竞争网络带宽。
**解决**：
1. 移除 CASE Session 建立时的批量 Reachable 通知（HomeKit 初始读取已覆盖）
2. CASE Session 去重窗口从 3 秒增至 30 秒，减少频繁重连时的流量尖峰

### Q21: 网关虚拟设备已移除

**原设计**：每个 LoRa 网关创建一个 OnOff Light Matter 端点，On=配对，Toggle=删除子设备。
**问题**：HomeKit 中出现多余的"灯开关"虚拟设备，用户体验差且不标准。
**解决**：已移除网关虚拟设备。配对通过 2击按键触发，删除所有子设备通过 3击按键触发。

## 十七、已知限制

1. **设备表容量**：默认最大 12 个桥接设备（可通过 menuconfig 调整，范围 1-12）
2. **无 TLS**：MQTT Broker 仅支持 TCP，未启用 TLS（节省内存）
3. **无持久化**：重启后设备表清空，需等待 LoRa 网关重新上报
4. **单核 broker**：Broker 固定在 core 1，Matter 在默认 core
5. **MQTT 认证**：用户名密码以明文传输（无 TLS），局域网内可接受
6. **涂鸦 App 动态端点**：涂鸦 App 对动态端点支持可能不完美，新设备添加后可能需要刷新或重启 App
7. **测试 VID**：默认使用 VID=0xFFF1（测试厂商 ID），涂鸦 App / Samsung SmartThings 会提示"未认证设备"。Apple HomeKit 和 Google Home 在测试 VID 下可正常配网使用。

## 附录 A：mDNS 方案版本记录

> 记录 mDNS 方案的演进决策，便于回退和排查。

### 当前方案（单实例 espressif/mdns）

**配置**：
- `CONFIG_USE_MINIMAL_MDNS=n`（sdkconfig.defaults）
- main.cpp 调用 `mdns_init()` + `mdns_hostname_set("matter-broker")`
- ESP32DnssdImpl.cpp L162-168 注释 `mdns_hostname_set(MAC_hex)`
- ESP32DnssdImpl.cpp L220 `mdns_service_subtype_add_for_host` 第 4 参数传 NULL

**工作机制**：单 mDNS 实例（espressif/mdns）负责全部功能
- Matter DNS-SD 服务发布（SRV/TXT/AAAA）：App 发现设备
- `matter-broker.local` A 查询响应：LoRa 网关解析 IP
- IGMP 让 espressif/mdns 加入组播组 224.0.0.251，收发 mDNS 包

**安全性验证**：
- `mdns_init()` 被 main.cpp 和 Matter SDK 的 `EspDnssdInit()` 各调用一次，第二次是幂等 no-op（内部检查 `mdns_priv_is_server_init()`），安全
- 所有服务操作（add/remove/subtype）统一使用全局 hostname `matter-broker`，无 mismatch
- SRV→A/AAAA 记录链路完整：服务 SRV 指向 `matter-broker.local`，全局 hostname 响应 A/AAAA 查询

> ESP-IDF v5.5.4 中 lwIP 的 IGMP 和组播发送由 lwipopts.h 硬编码启用（LWIP_IGMP=1），无需在 Kconfig 中配置。

### 已否决的替代方案（双实例并存，已证实不可用）

**配置**：
- `CONFIG_USE_MINIMAL_MDNS=y`

**否决原因**：minimal mdns 和 espressif/mdns 都绑定 UDP 5353，lwIP 不允许多个 PCB 绑定同一端口。espressif/mdns 先启动占用 5353，minimal mdns 后启动绑定失败，报错：
```
E (xxxx) chip[DIS]: Failed to initialize advertiser: 3000008  (CHIP_ERROR_INCORRECT_STATE)
```

### 回退步骤（如需从其他状态回到当前方案）

1. `sdkconfig.defaults`：`CONFIG_USE_MINIMAL_MDNS=n`
2. `main.cpp`：保留 `mdns_init()` + `mdns_hostname_set("matter-broker")`
3. `ESP32DnssdImpl.cpp` L162-168：确保 `mdns_hostname_set(service->mHostName)` 已注释
4. `ESP32DnssdImpl.cpp` L220：确保 `mdns_service_subtype_add_for_host` 第 4 参数为 NULL
5. `idf.py fullclean && idf.py build`
